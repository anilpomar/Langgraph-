Q1 Day 1 — fundamentals easy
"Can you explain what a node and an edge are in LangGraph, and how they work together?"
Hint: think about what each one does and how data moves between them
Strong answer
A node is a plain Python function that receives the current state dictionary, does some work, and returns a partial update to the state. It is the unit of work in a LangGraph application.

An edge is a connection between two nodes that tells LangGraph which node to run next. A fixed edge always goes to the same destination. A conditional edge uses a router function to pick the destination dynamically based on the current state.

Together they form a directed graph — nodes do the work, edges control the flow. Every LangGraph app is just nodes connected by edges sharing a common state dictionary.
# Node — does the work def greet(state): return {"message": "Hello!"} # Edge — controls the flow graph.add_edge("greet", "shout") # fixed graph.add_conditional_edges("greet", router, {...}) # dynamic

Q2 Day 1 — compile and invoke easy
"What is the difference between `graph.compile()` and `app.invoke()`? Can you explain what each one does?"
Hint: one prepares the graph, the other runs it
Strong answer
graph.compile() validates the graph definition — it checks that all nodes have edges, an entry point is set, and no connections are broken. It returns a compiled app object that is ready to run. Nothing executes at this stage.

app.invoke() actually runs the graph. You pass in the initial state dictionary and it executes every node in order, passing state between them, and returns the final state when it reaches END.

A good analogy: compile() is like building and checking a car before a race. invoke() is actually driving it.
app = graph.compile() # validates — nothing runs yet result = app.invoke({"message": "world"}) # runs the graph print(result) # {'message': 'HELLO, WORLD!'}

Q3 Day 2 — state and reducers medium
"What is a reducer in LangGraph? Why would you use `Annotated[list, operator.add]` instead of just `list` for a history field?"
Hint: think about what happens when two nodes both write to the same field
Strong answer
By default LangGraph replaces a state field with whatever a node returns. So if node1 writes ["step1"] and node2 writes ["step2"], the final value is ["step2"] — node1's data is lost.

A reducer changes this behaviour. Instead of replacing, LangGraph calls the reducer function to merge the old and new values. operator.add on lists does concatenation — so ["step1"] + ["step2"] gives ["step1", "step2"]. Nothing is lost.

You use Annotated[list, operator.add] whenever you want to accumulate values across nodes — chat history, logs, audit trails, or any growing list.
# Without reducer — node2 overwrites node1 history: list # node1 returns ["a"], node2 returns ["b"] # final: ["b"] ← "a" is gone # With reducer — both values kept history: Annotated[list, operator.add] # node1 returns ["a"], node2 returns ["b"] # final: ["a", "b"] ← both preserved

Q4 Day 3 — conditional edges medium
"What is the difference between `add_edge` and `add_conditional_edges`? When would you choose one over the other?"
Hint: think about fixed vs dynamic destinations
Strong answer
add_edge creates a fixed connection — after node A, always go to node B. You use it when the next step never changes regardless of what the data looks like.

add_conditional_edges creates a dynamic connection — after node A, call a router function with the current state, and the router's return value decides which node runs next. You use it when the next step depends on the data — like routing to approve, review, or reject based on a score.

In real applications, most graphs use both: fixed edges for predictable steps and conditional edges where business logic needs to branch.
# Fixed — always goes to shout after greet graph.add_edge("greet", "shout") # Dynamic — router decides based on state graph.add_conditional_edges( "evaluate", router, # called with current state {"approve": "approve_node", "reject": "reject_node"} )

Q5 Day 1–3 — scenario based hard
"You are building a loan approval system. Applications with a credit score above 750 get auto-approved, between 600–750 go to manual review, and below 600 get rejected. How would you design this in LangGraph?"
Hint: think about what goes in state, which nodes you need, and how the router works
Strong answer
I would design it with three layers. First the state — it needs a credit_score integer field and a decision string field to record the outcome. I would also add a history field with Annotated[list, operator.add] to track every step for auditing.

Second, the nodes — an intake node that validates the application, then three destination nodes: auto_approve, manual_review, and auto_reject. Each writes its decision to state.

Third, the router — after intake, a router function reads credit_score and returns "approve", "review", or "reject". The mapping dict connects those strings to the destination nodes via add_conditional_edges.
class LoanState(TypedDict): credit_score: int decision: str history: Annotated[list, operator.add] def router(state): s = state["credit_score"] if s > 750: return "approve" elif s >= 600: return "review" else: return "reject" g.add_conditional_edges( "intake", router, {"approve": "auto_approve", "review": "manual_review", "reject": "auto_reject"} )
